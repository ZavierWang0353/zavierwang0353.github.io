数据库基本概念

数据库依旧保留在内存，但是因为b+树和hash检索的原因速度比文件中要快

结构化查询语言(Structured Query Language)简称SQL

user/db/table_priv/columns_priv/host 权限表

数据查询语言DQL（Data Query Language）SELECT

数据操纵语言DML（Data Manipulation Language）INSERT，UPDATE，DELETE

数据控制功能DCL（Data Control Language）GRANT，REVOKE，COMMIT，ROLLBACK

约束：

  • NOT NULL: 不能为空。
  • UNIQUE: 不能重复，一个表允许有多个 Unique 约束。
  • PRIMARY KEY: 主键，一个表只允许出现一个。
  • FOREIGN KEY: 用于预防破坏表之间连接的动作，也能防止非法数据插入外键列，因为它必须是它指向的那个表中的值之一。
  • CHECK: 用于控制字段的值范围。

关联查询

交叉连接（CROSS JOIN）笛卡尔积

内连接（INNER JOIN）只选公共

外连接（LEFT JOIN/RIGHT JOIN）左主右空/左空右主

联合查询（UNION与UNION ALL）左右拼接，需要行数一样

全连接（FULL JOIN）两边空都包含（mysql不支持）

三范式

第一范式：每个列都不可以再拆分。（反例:元组列）

第二范式：在第一范式的基础上，非主键列完全依赖于主键，而不能是依赖于主键的一部分（针对联合主键）

第三范式：不能间接相关主键（不能依赖其他非主键）

  • 如何计算依赖

三范式有时候可以为性能牺牲一部分，view就是一个很好的例子

binlog录用方式

binlog的录入格式: statement, row(大规模变动很不划算), mixed(以statement为主, 无法使用再用row)

数据类型的trick

INT(10)只是表示显示范围,而存储内容并不会发生变化

VARCHAR更能节省空间, 但是频繁的修改可能会产生大量的碎片空间. Char适用于长度比较固定的字符串, 同时不容易产生碎片. 同时char快,属于空间换时间.

VARCHAR(50)和INT(50)一样都只表示

char(10)是真的10个字符

避免使用数字组成enum,可诱发混乱.

timestamp空间效率高于datetime, int存储时间戳不方便处理, 精确到微秒需要bigint

引擎

  • Innodb引擎：Innodb引擎提供了对数据库ACID事务的支持。并且还提供了行级锁和外键的约束。它的设计的目标就是处理大数据容量的数据库系统。
  • MyIASM引擎(原本Mysql的默认引擎)：不提供事务的支持，也不支持行级锁和外键。
  • MEMORY引擎：所有的数据都在内存中，数据的处理速度快，但是安全性不高。

MyISAM由三个文件定义, INNODB所有标都放在一个文件中

MyISAM易于一直,INNODB在数据量大的时候很困难.

MyISAM可以被压缩, INNODB的内存需求量较高

MYISAM数据和索引分别存储, INNODB是集中的

INNODB根据主键有序, MYISAM根据插入顺序

INNODB新增了对外键和事务的支持,行级锁定,

MYISAM更擅长select, INNODB擅长insert,update和delete

两个都是B+索引, MyISAM是堆, Innodb是索引组织表, INNODB支持哈希不支持全文索引,myISAM相反

INNODB是聚簇索引(索引和数据放在一起)MyISAM不是.

索引

将一列作为筛选条件时, 有索引和无索引的情况会相差非常多

  • 比较好奇创建索引的时候底层到底发生了什么,因为INNODB是一个聚簇索引所以应该索引和数据是放在一起的,但是印象中INNODB的存储会根据索引在子树上有序排列,想问下在实际内存中, 如何中和这两个索引的存储数据呢.

比如orderby就需要全局扫描, 并且写入内存中进行排序,这个是非常消耗性能的行为

Join在匹配之前也需要一个比对, 索引也能节省时间

如果要查询的字段都建立过索引，那么引擎会直接在索引表中查询而不会访问原始数据. 只要有一个字段没有建立索引就会做全表扫描

但是索引建多了也会耗费时间和占用磁盘空间,这个是一个需要折中的事情. 因此select后尽量只写必要的查询字段而不是select*

索引类型: 主键索引(唯一), 唯一索引(no null,可多个), 普通索引(可为null),全文索引

Hash索引适用于对等比较, 访问速度非常快, 因为只有一层映射,但是btree有多层

索引尽量选择基数大(value类型多的)值,否则效率会非常低下.(区分度的问题)

尽量使用短索引, 可考虑指定前缀长度. 前缀长度的选取参考平均匹配度(接近1即前缀几乎能确定一条记录)

组合索引非常重要的原则，mysql会一直向右匹配直到遇到范围查询(>、<、between、like)就停止匹配. 意即范围匹配最好放在最后

可在建立列或者后续用alter,create index添加索引.但是create index不可添加主键索引

B树的劣势在范围查询可能需要层序遍历,而b+只用考虑子节点.同时b树空间利用率高,一次读入节点信息多. 辅助索引存储的是主键值,这个和我之前好奇的矛盾是一样的么

关系型数据库的四个原则

ACID:原子(事务不可分割),一致(执行事务前后对同一数据读取时相同的),隔离性(并发时各个操作时独立的),持久性(对数据库更新要持久)

脏读: 读取更改前的数据

不可重复读: 多次查询中数据不一样

幻读: 两次查询数据量不一样

read-uncommitted:可读取尚未提交的数据变更

READ-COMMITTED: 允许读取并发事务已经提交的数据

REPEATABLE-READ:同一字段的多次读取结果都是一致的

SERIALIZABLE:所有的事务依次逐个执行，这样事务之间就完全不可能产生干扰

锁

INNODB使用索引来完成行锁

record lock:单个记录锁

gap lock: 锁定某个范围但不包括本身(repeatbale-read使用)

next-key lock: record+gap

解决死锁: 力度, 原子操作, 操作顺序

悲观乐观分别对应不同的业务场景: 改写多和读取多

视图

行和列数据来自定义视图的查询所引用基本表，并且在具体引用视图时动态生成。视图的列可以来自不同的表

简化sql查询，访问权限，更改数据格式和表示（逻辑独立性）

性能，修改限制

  • 游标到底是个啥

缺点：调试和移植问题，重新编译，维护问题

级联更改，维护困难

  • After Delete

杂项

因此，在不再需要一张表的时候，用drop；在想删除部分数据行时候，用delete；在保留表而删除所有数据的时候用truncate。

UNION与UNION ALL的区别？

  • 如果使用UNION ALL，不会合并重复的记录行
  • 效率 UNION 高于 UNION ALL

执行计划

低性能的SQL语句的定位。explain命令来查看语句的执行计划

type(非常重要，可以看到有没有走索引) 访问类型

  • ALL 扫描全表数据

  • index 遍历索引

  • range 索引范围查找

  • index_subquery 在子查询中使用 ref

  • unique_subquery 在子查询中使用 eq_ref

  • ref_or_null 对Null进行索引的优化的 ref

  • fulltext 使用全文索引

  • ref 使用非唯一索引查找数据

  • eq_ref 在join查询中使用PRIMARY KEYorUNIQUE NOT NULL索引关联。

  • 什么是全文索引

key_length 索引长度

ref 表示上述表的连接匹配条件，即哪些列或常量被用于查找索引列上的值

rows 返回估算的结果集数目，并不是一个准确的值。

extra 的信息非常丰富，常见的有：

  1. Using index 使用覆盖索引
  2. Using where 使用了用where子句来过滤结果集
  3. Using filesort 使用文件排序，使用非索引列进行排序时出现，非常消耗性能，尽量优化。
  • 使用了文件排序

大表数据查询，怎么优化

  1. 优化shema、sql语句+索引；
  2. 第二加缓存，memcached, redis；
  3. 主从复制，读写分离；
  4. 垂直拆分，根据你模块的耦合度，将一个大的系统分为多个小的系统，也就是分布式系统；
  5. 水平切分
  • 预测性查到内容
  • 延迟关联或子查询优化
